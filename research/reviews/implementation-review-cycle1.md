# Implementation Review -- Cycle 1

Reviewer assessment of the 10 summary documents for implementing the lambda cube
(and extensions to CIC/co-inductive constructions) in a proof assistant (Lean 4).

---

## Blocking Issues

### B1. No algorithmic type-checking specification for any system beyond STLC

The documents provide declarative typing rules for every system (STLC through CIC),
but the conversion rule is non-deterministic: it can be applied at any point in a
derivation. Document 02 (STLC) gives a bidirectional algorithm and document 09
(implementation techniques) describes bidirectional checking in general, but neither
gives the full algorithmic rules for a dependent type checker. Critically missing:

- An algorithmic formulation of the conversion rule for PTS/CoC/CIC. The documents
  say "normalize both sides and compare," but do not specify whether to use WHNF
  comparison with lazy unfolding (the standard approach) or full normalization. The
  distinction matters enormously for performance.
- The interaction between conversion checking and the Product/Abstraction rules in
  the PTS formulation. In a bidirectional algorithm, when do you switch from
  synthesis to checking mode in a dependent setting? Document 09 mentions Coquand's
  1996 algorithm but does not reproduce its rules.
- How to handle the Conversion rule algorithmically: do you always normalize the
  synthesized type and the expected type to WHNF and then structurally compare, or
  do you attempt unification? The documents are silent on this.

**What is needed:** A complete set of bidirectional algorithmic typing rules for a
PTS with two sorts, explicitly showing where conversion checks occur and what form
of normalization is used at each check point. Coquand's 1996 algorithm or the
elaboration-zoo approach should be spelled out, not merely cited.

### B2. Universe constraint solving is described only in passing

Document 07 (CIC) mentions universe constraints generated by inductive type
definitions and product formation rules, and document 06 (CoC) mentions universe
polymorphism (Sozeau and Tabareau, 2014), but there is no description of:

- How to represent universe level expressions (constants, max, +1, variables).
- The constraint language (e.g., `u <= v`, `u < v`, `u = max(v, w)`).
- An algorithm for checking satisfiability of a set of universe constraints.
- How to handle universe polymorphism: when to generalize, when to instantiate, how
  to carry constraint sets through elaboration.

This is a blocking issue because any implementation of CIC must solve universe
constraints, and getting this wrong leads to Girard's paradox.

**What is needed:** A specification of the universe constraint language, the
constraint-generation rules for each typing rule, and either a reference algorithm
or enough detail to derive one. The key reference is Sozeau and Tabareau (2014),
but the documents should at least reproduce the constraint-generation rules.

### B3. Guardedness checking for coinductive types lacks algorithmic detail

Document 08 (coinductive constructions) describes the guardedness condition
qualitatively: "each corecursive call must be directly under a constructor." But
there is no algorithm. Missing:

- A formal definition of the "guarded by constructors" predicate on terms,
  accounting for let-bindings, match expressions, and nested function calls.
- How to track guardedness through case analysis (the guardedness condition must
  interact with pattern matching).
- The distinction between guard-checking at the syntactic level (Coq's approach)
  and guard-checking via sized types (Agda's approach). The document describes both
  concepts but neither algorithm.

**What is needed:** An inductive definition of the guardedness predicate on CIC
terms, with rules covering all term formers, or a reference to Gimenez's 1995
formalization with enough detail to implement it.

### B4. Structural recursion checking is underspecified

Document 07 (CIC) describes fixpoint reduction and notes that "the fixpoint unfolds
only when its structurally decreasing argument is a constructor application." But
the documents do not specify:

- How to determine which argument is the "structurally decreasing" one.
- The algorithm for checking that all recursive calls are on structurally smaller
  arguments (the "guard condition" for fix).
- How structural decrease interacts with nested pattern matching, mutual recursion,
  and let-bindings.
- Whether the implementation should use a syntactic guard check or a more semantic
  approach (e.g., sized types, well-founded recursion via Acc).

**What is needed:** The guard-checking algorithm for `fix` definitions, specifying
how "structurally smaller" is tracked through pattern matching branches.

## Significant Gaps

### S1. The PTS unification is given but the parameterized implementation is not

Document 05 (lambda cube) provides the PTS typing rules and shows how all eight
systems are instances of a single parameterized framework with (S, A, R). This is
excellent for understanding the theory, but the documents do not bridge from this
to a concrete implementation:

- What Lean 4 data type should represent the unified term syntax? The pseudo-term
  grammar is given (`T ::= x | s | T T | lambda x:T.T | Pi x:T.T`), which is
  helpful, but there is no discussion of how to represent sorts, how to distinguish
  `Pi` from `lambda` in the implementation, or what annotations are needed.
- Should the implementation be parameterized by (S, A, R) at the type level (using
  Lean typeclasses or structure parameters), or should it be a single
  implementation that takes (S, A, R) as runtime data?
- The conversion rule requires normalization, but the normalization strategy may
  differ between systems (e.g., type-level normalization in System F-omega is
  trivially terminating, while full normalization in CoC requires more care).

**What is needed:** A concrete Lean 4 inductive type for PTS terms, a discussion
of parameterization strategy, and guidance on how the type-checking algorithm
should be structured to work across all eight systems.

### S2. Term representation choice is discussed but not decided

Document 09 (implementation techniques) provides an excellent survey of variable
representation strategies (named, de Bruijn indices, de Bruijn levels, locally
nameless, HOAS, PHOAS) with a comparison table. However, it does not make a
recommendation for the specific project at hand (formalization in Lean 4), nor does
it discuss:

- Which representation is best for proving metatheory in Lean 4? The locally
  nameless approach is noted as popular for Coq formalizations, but Lean 4
  formalizations may favor intrinsically-typed de Bruijn indices (as in PLFA).
- How to handle the two-level approach (indices for syntax, levels for NbE values)
  in Lean 4 specifically.
- Concrete Lean 4 inductive type definitions for any of these representations.

**What is needed:** A concrete recommendation and a skeleton Lean 4 data type for
terms, contexts, and substitutions, with justification for the choice.

### S3. Strict positivity checking algorithm is incomplete

Document 07 (CIC) gives a clear inductive definition of the strict positivity
condition, including the nested positivity clause. This is close to implementable,
but missing:

- How to implement the check as a recursive function on types. The definition is
  given as a predicate, not as an algorithm that traverses constructor types.
- How to handle the "nested positivity" case, which requires looking up the
  constructors of a previously defined inductive type and checking them against the
  new type being defined. This requires access to a global environment of inductive
  definitions.
- How parameters are substituted when checking nested positivity.

**What is needed:** Pseudocode or a Lean 4 function definition for the strict
positivity check.

### S4. Iota-reduction and pattern matching compilation are underspecified

Document 07 (CIC) gives the iota-reduction rule for pattern matching and the
fixpoint unfolding rule. However:

- The typing rule for `match` expressions is given informally ("the return clause
  specifies the result type") but the full dependent pattern matching typing rule
  is not spelled out with enough precision to implement.
- There is no discussion of how to compile complex pattern matching (nested
  patterns, or-patterns, overlapping patterns) into the kernel's primitive `match`.
- The interaction between pattern matching and fixpoint reduction (when does a
  fixpoint unfold? only when the scrutinee is constructor-headed?) is stated but
  not formalized as a reduction rule on the full term syntax.

**What is needed:** The complete typing rule for `match` with dependent return
type, and the reduction rules for `match` and `fix` as they would appear in a
small-step operational semantics.

### S5. Cumulativity is mentioned but not formalized

Documents 06 and 07 mention universe cumulativity (`Type(i) <= Type(j)` when
`i <= j`) but do not specify:

- Whether cumulativity is a subtyping relation or a coercion.
- How cumulativity interacts with the conversion rule: is `A : Type(0)` convertible
  with `A : Type(1)`, or does conversion checking need a separate subtyping check?
- The cumulativity rule for inductive types: is `List : Type(0) -> Type(0)` a
  subtype of `List : Type(1) -> Type(1)`?

**What is needed:** Formal cumulativity rules showing how subtyping integrates
with conversion checking.

## Improvements

### I1. NbE algorithm is described well but not for dependent types

Document 09 gives a clear NbE algorithm for STLC with semantic domain, evaluation,
and reification. It mentions that "the approach extends naturally" to dependent
types but does not give the extended algorithm. For dependent types, the semantic
domain must include:

- Universe values
- Pi-type values (closures for the codomain)
- Neutral types (stuck type computations)
- Inductive type values and constructor values

The NbE approach for dependent types is well-known but nontrivial. A complete
algorithm would significantly ease implementation.

### I2. Testing and validation guidance is absent

None of the 10 documents provide:

- Test cases for type-checking implementations (e.g., "this term should type-check
  at this type," "this term should fail to type-check").
- Known tricky examples that implementations get wrong.
- A regression test suite or benchmark.
- Guidance on how to validate that an implementation is correct beyond proving it so.

**What is needed:** A catalog of test cases organized by system, covering:
- Basic well-typed terms (identity, Church encodings)
- Terms that should be rejected (ill-typed, fails positivity, fails guardedness)
- Edge cases (universe level corner cases, Hurkens' paradox as a soundness check)
- Performance-sensitive examples (deeply nested types, large terms)

### I3. Proof strategy for correctness is described at a high level only

Document 10 (metatheory) gives an excellent survey of proof techniques (logical
relations, reducibility candidates, parallel reduction for confluence) but does not
provide:

- A roadmap for which properties to prove first and in what order.
- Guidance on which properties are feasible to formalize in Lean 4 vs. which
  should be deferred.
- Lean 4-specific advice (e.g., how to handle well-founded recursion in
  normalization proofs, how to use Lean's `termination_by` for structural recursion
  on typing derivations).
- The expected difficulty level of each proof. For instance, strong normalization
  for CoC is known to be very hard to formalize; is it realistic to attempt it, or
  should the project rely on extrinsic evidence?

**What is needed:** A prioritized proof roadmap, such as:
1. Prove progress and preservation for STLC (warm-up).
2. Prove confluence via parallel reduction (applicable to all systems).
3. Prove strong normalization for STLC via Tait's method.
4. Extend to System F via reducibility candidates.
5. Defer CoC/CIC strong normalization to later cycles.

### I4. Eta-equality treatment is inconsistent across documents

Different documents handle eta differently:
- Document 01 (untyped) introduces eta-reduction.
- Document 06 (CoC) mentions eta-conversion as "part of definitional equality in
  Coq."
- Document 08 (coinductive) notes that eta for coinductive types is "computationally
  problematic."
- Document 09 (implementation) mentions NbE variants that produce beta-normal vs.
  beta-eta-normal forms.

But there is no unified statement of when eta should be included in the conversion
relation for the implementation. For a kernel implementation, this matters: if the
conversion checker includes eta, it must handle eta for functions, pairs (if
present), and unit types.

**What is needed:** A clear decision on whether the implementation should use
beta-only or beta-eta conversion, with the conversion-checking algorithm adjusted
accordingly.

### I5. Elaboration from surface syntax is not covered

The documents describe fully explicit core calculi where every lambda has a type
annotation. But any usable implementation needs:

- Implicit argument inference (metavariables + unification).
- Pattern unification for solving metavariables (described in document 09 but not
  connected to the other systems).
- How to elaborate user-written definitions into core CIC terms.

This is arguably out of scope for a kernel implementation, but document 09's
treatment of elaboration is disconnected from the rest of the documents.

---

## Summary Assessment

### What is well-covered:

- **Declarative specifications** for all systems from untyped lambda calculus through
  CIC are thorough and precise. The typing rules, reduction rules, and kinding rules
  are complete enough to understand each system.
- **The PTS formulation** in document 05 is clear and directly supports a unified
  parameterized implementation.
- **Variable representation** in document 09 is surveyed comprehensively.
- **Metatheory** in document 10 provides a solid conceptual foundation for proofs.
- **Historical context and references** are excellent throughout; an implementer who
  gets stuck can find the original sources.

### What is critically missing:

1. An **algorithmic type-checking specification** for dependent types (the gap
   between declarative rules and a runnable algorithm).
2. **Universe constraint solving** algorithm.
3. **Guard-checking algorithms** for structural recursion and productivity.
4. **Test cases** for validating implementations.
5. A **concrete Lean 4 data type** for terms and a parameterization strategy.

### Bottom line:

An experienced implementer familiar with dependent type theory could implement a
type checker for the pure lambda cube (without inductive types) from these documents,
supplemented by Coquand's 1996 algorithm paper and the elaboration-zoo repository.
Extending to CIC would require additional references for positivity checking,
pattern matching, universe constraint solving, and guard checking. Extending to
coinductive types would require Gimenez's 1995 paper or Abel's work on sized types.

The documents are strongest as **specifications** and **conceptual introductions**.
They are weakest as **implementation guides**. A second research cycle should focus
on the algorithmic gap: converting declarative specifications into deterministic
algorithms, and providing concrete data structures and test cases.
