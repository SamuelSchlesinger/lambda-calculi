# System F: The Polymorphic Lambda Calculus

## Overview

System F, also known as the **polymorphic lambda calculus**, the **second-order lambda calculus**, or **lambda-2** (λ2), is a typed lambda calculus that extends the simply typed lambda calculus (STLC) with **universal quantification over types**. This single extension---adding type variables, type abstraction, and type application---yields a dramatically more expressive system that can encode booleans, natural numbers, pairs, lists, and existential types directly, without primitive data types.

System F occupies a central position in the theory of programming languages. It is the formal foundation for **parametric polymorphism**, the mechanism underlying generic programming in languages such as Haskell, ML, Scala, and Rust. Under the Curry-Howard correspondence, System F corresponds to **second-order intuitionistic propositional logic**.

---

## 1. Historical Context

### 1.1 Jean-Yves Girard (1972)

System F was invented by **Jean-Yves Girard** in his 1972 PhD thesis, *"Interprétation fonctionnelle et élimination des coupures de l'arithmétique d'ordre supérieur"* (Functional interpretation and cut elimination in higher-order arithmetic), completed at Université Paris VII under the supervision of Jean-Louis Krivine.

Girard's motivation was **proof-theoretic**: he sought a functional interpretation (in the Brouwer-Heyting-Kolmogorov style) of second-order arithmetic. System F emerged as the term calculus corresponding to second-order intuitionistic propositional logic. A major achievement of the thesis was the proof of **strong normalization** for System F using the method of *candidats de réductibilité* (reducibility candidates).

Girard also proved a **Representation Theorem**: every function on natural numbers that can be proved total in second-order intuitionistic predicate logic (P2) can be represented as a term in System F (F2). This establishes a precise connection between provability in logic and definability in the calculus.

### 1.2 John C. Reynolds (1974)

**John C. Reynolds** independently discovered the same formal system in his 1974 paper *"Towards a theory of type structure."* Reynolds' motivation was entirely different from Girard's: he sought to build a type system for a programming language that could express **parametric polymorphism**---the idea that functions and data structures should be definable generically, without dependence on specific types.

Reynolds proposed allowing types to be passed as a special kind of parameter, with usage restricted to permit syntactic checking of type correctness. This perspective directly influenced the design of polymorphic programming languages.

### 1.3 The Girard-Reynolds Isomorphism

Despite their independent discovery from different fields (proof theory vs. programming languages), the two formulations are isomorphic. The **Girard-Reynolds isomorphism** formalizes this:

- Girard's Representation Theorem yields a *projection* from second-order logic (P2) into System F (F2).
- Reynolds' Abstraction Theorem yields an *embedding* of System F into second-order logic.
- The composition (Girard projection followed by Reynolds embedding) is the identity.

This isomorphism is a striking instance of the deep unity between logic and computation.

## 2. Syntax

### 2.1 Types

The types of System F are generated by the following grammar:

```
τ, σ ::= α                   (type variable)
       | τ → σ               (function/arrow type)
       | ∀α. τ               (universal quantification)
```

where α ranges over a countably infinite set of **type variables**.

Key points:
- Type variables α, β, γ, ... range over types and can be bound by the universal quantifier ∀.
- The quantifier ∀α binds α in τ within ∀α. τ, and types are identified up to **alpha-equivalence** of bound type variables.
- The scope of ∀ extends as far to the right as possible: ∀α. α → α means ∀α. (α → α).
- The arrow → associates to the right: τ → σ → ρ means τ → (σ → ρ).

### 2.2 Terms

The terms of System F extend those of the simply typed lambda calculus with type abstraction and type application:

```
t, s ::= x                   (variable)
       | λx:τ. t             (term abstraction)
       | t s                 (term application)
       | Λα. t               (type abstraction)
       | t [τ]               (type application)
```

where x ranges over term variables and α over type variables.

Key points:
- **Term abstraction** λx:τ. t is the same as in STLC, binding term variable x with type annotation τ.
- **Type abstraction** Λα. t (capital lambda) abstracts over a type variable α in the term t. This is a term-level construct that introduces polymorphism.
- **Type application** t [τ] instantiates a polymorphic term t at a specific type τ. Square brackets are conventional but sometimes omitted.

**Example**: The polymorphic identity function:
```
id = Λα. λx:α. x
```
This has type ∀α. α → α. Applying it to a specific type: id [Nat] : Nat → Nat.

### 2.3 Contexts

A typing context Γ is a sequence of variable-type bindings:

```
Γ ::= ∅ | Γ, x:τ | Γ, α
```

where α declarations indicate that α is a valid type variable in scope.

## 3. Typing Rules

The typing judgment has the form **Γ ⊢ t : τ**, meaning "in context Γ, term t has type τ." The rules are:

### Variable
```
  x:τ ∈ Γ
  ─────────── (Var)
   Γ ⊢ x : τ
```

### Term Abstraction (→-Introduction)
```
   Γ, x:τ ⊢ t : σ
  ─────────────────────── (Abs)
   Γ ⊢ (λx:τ. t) : τ → σ
```

### Term Application (→-Elimination)
```
   Γ ⊢ t : τ → σ    Γ ⊢ s : τ
  ──────────────────────────────── (App)
         Γ ⊢ t s : σ
```

### Type Abstraction (∀-Introduction)
```
       Γ, α ⊢ t : τ
  ──────────────────────── (TAbs)
   Γ ⊢ (Λα. t) : ∀α. τ
```

where α must not appear free in Γ (the standard freshness/eigenvariable condition).

### Type Application (∀-Elimination)
```
   Γ ⊢ t : ∀α. τ
  ───────────────────────── (TApp)
   Γ ⊢ t [σ] : τ[σ/α]
```

where τ[σ/α] denotes the capture-avoiding substitution of σ for α in τ.

## 4. Reduction Rules

System F has two forms of beta reduction:

### Term-level beta reduction (β)
```
(λx:τ. t) s  ⟶_β  t[s/x]
```
The standard beta reduction from lambda calculus, substituting the argument s for the bound variable x in the body t.

### Type-level beta reduction (β_τ)
```
(Λα. t) [σ]  ⟶_β  t[σ/α]
```
Type-level beta reduction substitutes the type argument σ for the bound type variable α throughout the term t.

### Congruence and Compatibility

Reduction applies under all term formers (congruence/compatibility rules):

```
t ⟶ t'  implies  t s ⟶ t' s
t ⟶ t'  implies  s t ⟶ s t'
t ⟶ t'  implies  λx:τ. t ⟶ λx:τ. t'
t ⟶ t'  implies  Λα. t ⟶ Λα. t'
t ⟶ t'  implies  t [σ] ⟶ t' [σ]
```

### Eta Rules (optional)

Some presentations also include eta rules:
```
λx:τ. t x  ⟶_η  t       (when x ∉ FV(t))
Λα. t [α]  ⟶_η  t       (when α ∉ FTV(t))
```

### Type Erasure

An important property: types can be erased without affecting computational behavior. The erasure function |·| maps System F terms to untyped lambda terms:
```
|x|           = x
|λx:τ. t|    = λx. |t|
|t s|         = |t| |s|
|Λα. t|      = |t|
|t [τ]|      = |t|
```

The **adequacy theorem** states that if t ⟶* v in System F, then |t| ⟶* |v| in untyped lambda calculus.

## 5. Church Encodings in System F

One of the remarkable features of System F is that many data types can be encoded directly using only the primitives of the calculus (type abstraction, type application, term abstraction, term application). Unlike the untyped lambda calculus, these encodings can be given precise types.

### 5.1 Booleans

```
Bool = ∀α. α → α → α

true  = Λα. λx:α. λy:α. x       : Bool
false = Λα. λx:α. λy:α. y       : Bool
```

A boolean is a function that takes two arguments (one for the true case, one for the false case) and returns one of them. The conditional is simply application:

```
if : Bool → ∀α. α → α → α
if b = b
```

Boolean operations:
```
not = λb:Bool. Λα. λx:α. λy:α. b [α] y x
and = λa:Bool. λb:Bool. Λα. λx:α. λy:α. a [α] (b [α] x y) y
or  = λa:Bool. λb:Bool. Λα. λx:α. λy:α. a [α] x (b [α] x y)
```

### 5.2 Natural Numbers (Church Numerals)

```
Nat = ∀α. (α → α) → α → α

zero = Λα. λs:(α → α). λz:α. z                    : Nat
succ = λn:Nat. Λα. λs:(α → α). λz:α. s (n [α] s z) : Nat → Nat
```

The numeral n is a function that applies its first argument (a successor function) exactly n times to its second argument (a zero element):

```
one   = Λα. λs:(α → α). λz:α. s z
two   = Λα. λs:(α → α). λz:α. s (s z)
three = Λα. λs:(α → α). λz:α. s (s (s z))
```

Arithmetic operations:
```
plus  = λm:Nat. λn:Nat. Λα. λs:(α → α). λz:α. m [α] s (n [α] s z)
times = λm:Nat. λn:Nat. Λα. λs:(α → α). λz:α. m [α] (n [α] s) z
```

### 5.3 Pairs (Products)

```
Pair = λα. λβ. ∀γ. (α → β → γ) → γ

pair = Λα. Λβ. λx:α. λy:β. Λγ. λf:(α → β → γ). f x y
       : ∀α. ∀β. α → β → Pair α β

fst = Λα. Λβ. λp:(Pair α β). p [α] (λx:α. λy:β. x)
       : ∀α. ∀β. Pair α β → α

snd = Λα. Λβ. λp:(Pair α β). p [β] (λx:α. λy:β. y)
       : ∀α. ∀β. Pair α β → β
```

### 5.4 Lists

```
List = λα. ∀β. (α → β → β) → β → β

nil  = Λα. Λβ. λc:(α → β → β). λn:β. n
       : ∀α. List α

cons = Λα. λh:α. λt:(List α). Λβ. λc:(α → β → β). λn:β. c h (t [β] c n)
       : ∀α. α → List α → List α
```

A list is its own fold: given a "cons" function and a "nil" value, it reduces itself by replacing each cons with the given function and nil with the given value.

### 5.5 Sum Types (Disjoint Unions)

```
Either = λα. λβ. ∀γ. (α → γ) → (β → γ) → γ

left  = Λα. Λβ. λx:α. Λγ. λl:(α → γ). λr:(β → γ). l x
right = Λα. Λβ. λy:β. Λγ. λl:(α → γ). λr:(β → γ). r y
```

### 5.6 The Void / Empty Type

```
Void = ∀α. α
```

There is no closed term of this type (in a consistent/normalizing system), corresponding to the logical falsum ⊥.

### 5.7 The Unit Type

```
Unit = ∀α. α → α

unit = Λα. λx:α. x   : Unit
```

## 6. Existential Types and Data Abstraction

### 6.1 Encoding Existential Types

Existential types are not primitive in System F but can be encoded via the following Church encoding:

```
∃α. τ  ≡  ∀β. (∀α. τ → β) → β
```

This encoding captures the essence of data abstraction: an existential package hides the representation type α while exposing operations of type τ.

### 6.2 Pack and Unpack

The introduction form (**pack**) creates an existential package by providing a witness type and implementation:

```
pack [σ, t] as ∃α. τ  ≡  Λβ. λf:(∀α. τ → β). f [σ] t
```

The elimination form (**unpack**) opens an existential package:

```
unpack e as [α, x] in t  ≡  e [σ_result] (Λα. λx:τ. t)
```

where σ_result is the type of the overall expression.

### 6.3 Abstract Types Have Existential Type

Mitchell and Plotkin (1988) established the foundational connection between abstract data types in programming languages and existential types. They showed that abstract type declarations in languages like CLU, Ada, and ML can be formalized as existential types.

**Example**: A counter ADT:

```
CounterADT = ∃α. { new : α, increment : α → α, read : α → Nat }
```

An implementation using natural numbers:

```
natCounter = pack [Nat, { new = 0, increment = succ, read = λn:Nat. n }]
             as CounterADT
```

The representation type (Nat) is hidden from clients, who can only interact with the counter through the provided operations.

## 7. Strong Normalization

### 7.1 Statement

**Theorem (Girard, 1972)**: Every well-typed term in System F is **strongly normalizing** under β-reduction (both term-level and type-level). That is, every reduction sequence starting from a well-typed term is finite.

This is a much stronger property than weak normalization (which only guarantees that *some* reduction sequence terminates). It implies:
- Every well-typed System F term has a unique normal form (by confluence + strong normalization).
- System F cannot encode general recursion; it is not Turing-complete.
- The set of functions Nat → Nat definable in System F is precisely those provably total in second-order Peano arithmetic.

### 7.2 Girard's Proof: Reducibility Candidates

The proof of strong normalization for System F is substantially more difficult than for the simply typed lambda calculus. A naive induction on typing derivations fails because the type application rule (∀-elimination) substitutes types into other types, potentially making them larger and destroying the inductive structure.

Girard's solution uses the method of **reducibility candidates** (*candidats de réductibilité*), extending Tait's method of logical predicates/computability.

**Definition**: A set S of terms is a **reducibility candidate** (for a given type) if it satisfies:

1. **(CR1)**: Every term in S is strongly normalizing.
2. **(CR2)**: S is closed under reduction: if t ∈ S and t ⟶ t', then t' ∈ S.
3. **(CR3)**: S is closed under "neutral expansion": if t is a *neutral* term (i.e., not a lambda abstraction or type abstraction) and all its one-step reducts are in S, then t ∈ S.

Let **CR** denote the set of all reducibility candidates.

**Interpretation of types**: Types are interpreted as reducibility candidates:

- ⟦α⟧ξ = ξ(α), where ξ is a valuation mapping type variables to reducibility candidates.
- ⟦τ → σ⟧ξ = { t | for all s ∈ ⟦τ⟧ξ, t s ∈ ⟦σ⟧ξ }
- ⟦∀α. τ⟧ξ = ⋂_{S ∈ CR} ⟦τ⟧ξ[α↦S]

The key steps of the proof:

1. Show that the interpretation of every type is itself a reducibility candidate (closure properties).
2. Prove the **Fundamental Lemma** (Adequacy): If Γ ⊢ t : τ, then under any substitution of terms from the appropriate reducibility candidates for the free variables, the result lies in ⟦τ⟧.
3. Conclude strong normalization: since every reducibility candidate consists only of strongly normalizing terms (CR1), every well-typed term is strongly normalizing.

### 7.3 Significance of the Proof Technique

The reducibility candidates method (in its biorthogonal form) remains the **only known technique** for proving strong normalization in second-order settings. It has been adapted and extended for many subsequent systems, including:
- System Fω (higher-order polymorphism)
- The Calculus of Constructions
- Various extensions with effects, linear types, etc.

The proof cannot be carried out within second-order arithmetic itself---it requires principles from higher-order arithmetic, reflecting Gödel's incompleteness phenomenon.

## 8. Parametricity

### 8.1 Reynolds' Abstraction Theorem (1983)

**John C. Reynolds** (1983) defined a **relational interpretation** of System F types and proved the **Abstraction Theorem**: every well-typed term, when evaluated in related environments, produces related results.

#### Relational Interpretation

Given types τ and σ interpreted as sets A and B, a relation R ⊆ A × B is constructed inductively on type structure:

- **Type variable α**: The relation is given by a relational environment ρ(α) = R.
- **Arrow type τ₁ → τ₂**: Given relations R₁ for τ₁ and R₂ for τ₂, the relation on functions is:
  ```
  f R₁→R₂ g  iff  for all (a, b) ∈ R₁, (f(a), g(b)) ∈ R₂
  ```
- **Universal type ∀α. τ**: The relation is:
  ```
  x R_{∀α.τ} y  iff  for all sets A, B and relations R ⊆ A × B,
                       (x_A, y_B) ∈ ⟦τ⟧[α ↦ (A, B, R)]
  ```

#### Statement

**Abstraction Theorem**: If Γ ⊢ t : τ, then for any two environments η₁, η₂ related by the relational interpretation of Γ, the interpretations ⟦t⟧η₁ and ⟦t⟧η₂ are related by the relational interpretation of τ.

### 8.2 Wadler's "Theorems for Free!" (1989)

Philip Wadler popularized the consequences of parametricity by showing how to derive "free theorems"---properties of functions derivable purely from their types, without knowledge of their implementation.

#### Methodology

Given a polymorphic type, one instantiates the parametricity relation with specific relations (typically function graphs) to derive equational properties.

#### Examples of Free Theorems

**Example 1**: For any function `r : ∀α. α → α`:
```
r = id
```
The identity function is the *only* inhabitant of this type. The parametricity relation forces r to return its argument unchanged, since it has no way to construct or inspect values of an arbitrary type.

**Example 2**: For any function `r : ∀α. [α] → [α]` and any function `f : A → B`:
```
map f ∘ r = r ∘ map f
```
This says that r must commute with mapping: it can only rearrange, duplicate, or drop list elements, and the rearrangement pattern must be independent of the element values.

**Example 3**: For any function `r : ∀α. (α → α → α) → [α] → α` (a fold-like function) and `f : A → B`:
If `f (x ⊕ y) = f x ⊗ f y` for all x, y, then:
```
f (r ⊕ xs) = r ⊗ (map f xs)
```

**Example 4**: For the type `∀α. ∀β. (α → β) → α → β`:
Any inhabitant must be the application function: `r f x = f x`.

### 8.3 Dinaturality

Free theorems can also be understood categorically. Parametric polymorphic functions correspond to **dinatural transformations** (or, in many cases, natural transformations) between functors. The free theorem is precisely the dinaturality condition.

## 9. Curry-Howard Correspondence

System F corresponds to **second-order intuitionistic propositional logic** under the Curry-Howard isomorphism:

| System F | Second-Order Logic |
|---|---|
| Type τ | Proposition P |
| Term t : τ | Proof of P |
| Arrow type τ → σ | Implication P ⊃ Q |
| Universal type ∀α. τ | Universal quantification ∀X. P(X) |
| Type abstraction Λα. t | ∀-introduction |
| Type application t [τ] | ∀-elimination |
| Term abstraction λx:τ. t | ⊃-introduction (assumption discharge) |
| Term application t s | ⊃-elimination (modus ponens) |
| β-reduction | Cut elimination |
| Normal form | Cut-free proof |

Key consequences:

- **Strong normalization** of System F corresponds to the **cut elimination theorem** for second-order intuitionistic propositional logic.
- Church-encoded data types correspond to their logical counterparts:
  - `∀α. α → α → α` corresponds to the disjunction P ∨ P (trivially provable).
  - `∀α. α` corresponds to ⊥ (falsum), which has no proof.
  - `∃α. τ` (encoded as `∀β. (∀α. τ → β) → β`) corresponds to existential quantification.
- Girard's System F was explicitly motivated by this correspondence: he sought a computational interpretation of second-order logic proofs.

## 10. Type Inference and Decidability

### 10.1 Church-style vs. Curry-style

There are two presentations of System F:

- **Church-style** (à la Church): Type abstractions and type applications are explicit in terms. Type *checking* (verifying Γ ⊢ t : τ for a given annotated term t) is decidable.
- **Curry-style** (à la Curry): Terms are those of the untyped lambda calculus; type abstractions and applications are implicit. Both type checking and type inference (typability) are undecidable.

### 10.2 Wells' Undecidability Results (1999)

J. B. Wells (1999) proved two landmark results for Curry-style System F:

1. **Type checking is undecidable**: Given an untyped term t and a type τ, determining whether t can be assigned type τ is undecidable. Proved by reduction from **semi-unification** (the combination of first-order unification and first-order matching), which Kfoury, Tiuryn, and Urzyczyn showed to be undecidable.

2. **Typability is undecidable**: Given an untyped term t, determining whether there exists *any* type τ such that t is typable is undecidable. Proved by reduction from type checking.

3. **Equivalence**: Type checking and typability are computationally equivalent for System F.

These results were the first to resolve the decidability status of these problems for System F, which had been open since the 1970s.

### 10.3 Partial Type Inference and Bidirectional Approaches

Given the undecidability of full type inference, several approaches to partial type inference have been developed:

- **Local type inference** (Pierce and Turner, 2000): Uses bidirectional type checking where types flow from known positions to infer unknown ones, requiring annotations only at specific syntactic positions.

- **Bidirectional type checking** (Dunfield and Krishnaswami, 2013): Their system "Complete and Easy Bidirectional Typechecking for Higher-Rank Polymorphism" provides a practical algorithm for higher-rank polymorphism that is both sound and complete relative to a declarative specification, and decidable.

- **HMF** (Leijen): Simple type inference for first-class polymorphism, combining Hindley-Milner inference with limited support for impredicative instantiation.

- **MLF** (Le Botlan and Rémy): A more expressive system that extends ML-style inference with first-class polymorphism while retaining principal types.

## 11. Predicative vs. Impredicative Polymorphism

### 11.1 Impredicativity in System F

System F is **impredicative**: a universally quantified type ∀α. τ quantifies over *all* types, including ∀α. τ itself. This means polymorphic types can be instantiated with other polymorphic types:

```
id [∀β. β → β]  :  (∀β. β → β) → (∀β. β → β)
```

The self-application `(Λα. λx:α. x) [∀α. α → α] (Λα. λx:α. x)` is typable, which is impossible in the simply typed lambda calculus. This is the hallmark of impredicativity: definitions can refer to the totality being defined.

Impredicativity is essential for:
- Church encodings of data types (the encoded type quantifies over all types, including itself)
- Encoding existential types within System F
- The full expressive power of second-order quantification

### 11.2 Predicative Polymorphism

In a **predicative** system, type variables can only be instantiated with **monomorphic** types (types not containing ∀). This stratification avoids the circularity of impredicative definitions.

Predicative polymorphism is strictly weaker than impredicative polymorphism but has better meta-theoretic properties, particularly for type inference.

### 11.3 Stratification and the Ramified Hierarchy

Leivant (1983) introduced a stratified version of System F based on **ranks** of type quantifiers, measuring their nesting depth inside function type constructors:

- **Rank 0**: Monomorphic types (no quantifiers)
- **Rank 1**: ∀ appears only at the outermost level (prenex form): ∀α₁...∀αₙ. τ where τ is monomorphic
- **Rank k**: ∀ may appear to the left of at most k-1 arrows

## 12. Rank-n Polymorphism

### 12.1 Definition

A type has **rank n** if all universal quantifiers in it appear at depth at most n when counting left-hand sides of arrows:

- **Rank 0**: No quantifiers. Example: `Int → Bool`
- **Rank 1** (prenex): Quantifiers at the top only. Example: `∀α. α → α`
- **Rank 2**: Quantifiers may appear to the left of one arrow. Example: `(∀α. α → α) → Int`
- **Rank 3**: Example: `((∀α. α → α) → Int) → Bool`

### 12.2 Type Inference Properties

The decidability of type inference varies with rank:

| Rank | Type Inference | Notes |
|------|---------------|-------|
| Rank 1 (prenex) | Decidable, principal types | Hindley-Milner |
| Rank 2 | Decidable | Kfoury and Wells (1999) |
| Rank 3+ | Undecidable | Wells (1999) |
| Unrestricted (System F) | Undecidable | Wells (1999) |

This hierarchy shows that rank-2 polymorphism is the "sweet spot" where full type inference is still possible but significantly more expressive than rank-1.

### 12.3 Practical Applications

Rank-2 polymorphism is used in Haskell for:
- The `runST` function: `runST :: (∀s. ST s a) → a`, which uses the rank-2 type to ensure that mutable state does not escape the ST computation.
- Existential type wrappers.
- Lens libraries and various design patterns.

GHC Haskell supports arbitrary-rank polymorphism via the `RankNTypes` extension, using bidirectional type checking with programmer-supplied annotations.

## 13. System F and Intersection Types

### 13.1 Relationship

Intersection types and System F's universal quantification are related but distinct mechanisms for expressing polymorphism:

- **System F (∀)**: Parametric polymorphism---a single term works uniformly for all types.
- **Intersection types (∩)**: A term can have multiple types simultaneously; `t : τ₁ ∩ τ₂` means t has both type τ₁ and type τ₂.

### 13.2 Expressiveness

Intersection types are in some ways more expressive than System F:
- In the untyped lambda calculus with intersection types, a term is typable if and only if it is strongly normalizing (the Coppo-Dezani theorem).
- Intersection types can type some terms that System F cannot (they allow non-uniform behavior across instances).

However, System F provides **parametricity guarantees** that intersection types do not. A function of type ∀α. α → α *must* be the identity, but a function of type (Int → Int) ∩ (Bool → Bool) could behave differently on integers and booleans.

### 13.3 Combinations

Pierce (1991) studied the combination of intersection types with bounded polymorphism (System F with subtyping, known as System F<:). These combined systems provide a type-theoretic foundation for object-oriented programming with multiple inheritance.

Modern languages like TypeScript, Flow, and Scala combine parametric polymorphism with intersection and union types.

## 14. Denotational Semantics

### 14.1 Reynolds' Impossibility Result (1984)

Reynolds (1984) proved a fundamental negative result: **there is no set-theoretic model of System F** in which:
- Types are interpreted as sets
- The function type τ → σ is interpreted as the full function space (all functions from the set interpreting τ to the set interpreting σ)
- The universal type ∀α. τ is interpreted as an intersection over all sets

The core difficulty is **cardinality**: the universal type ∀α. τ would need to range over all sets, but by Cantor's theorem, the collection of all sets is not itself a set. More precisely, interpreting ∀α. α → α as ∏_{S ∈ Set} (S → S) yields a set that is too large to be an element of the domain being quantified over.

This result showed that naive set-theoretic semantics is fundamentally inadequate for impredicative polymorphism.

### 14.2 Alternative Models

Following Reynolds' impossibility result, several alternative approaches to modeling System F have been developed:

1. **Domain-theoretic models**: Using Scott domains or other domain-theoretic structures where function spaces are restricted to continuous (or stable) functions rather than all functions. Bruce, Meyer, and Mitchell (1990) constructed such models.

2. **Coherent spaces** (Girard): Girard's own approach using coherent spaces and linear logic provides models of System F.

3. **PER models** (Partial Equivalence Relations): Types are interpreted as PERs on a universal domain (such as the natural numbers or a universal lambda-algebra). This approach, developed by Longo, Moggi, and others, provides a natural model.

4. **Categorical models**: Seely (1987) showed that models of System F can be described as certain hyperdoctrines. The categorical approach using fibrations and indexed categories provides abstract frameworks for constructing models.

5. **Realizability models**: Using Kleene realizability or related notions, where types are interpreted as sets of realizers (codes for computable functions).

## 15. Relationship to ML-Style Polymorphism (Hindley-Milner)

### 15.1 Hindley-Milner as a Restriction of System F

The **Hindley-Milner** (HM) type system, used in ML, OCaml, and Haskell 98, is a carefully chosen restriction of System F that makes type inference decidable and practical:

| Feature | System F | Hindley-Milner |
|---------|----------|----------------|
| Polymorphism | Explicit (Λα, t[τ]) | Implicit (inferred) |
| Quantifier position | Anywhere in a type | Top-level only (rank-1/prenex) |
| Polymorphism introduction | Anywhere (Λα. t) | Only at let-bindings |
| Type annotations | Required (Church-style) | Optional (inferred) |
| Type inference | Undecidable | Decidable (Algorithm W) |
| Principal types | Not always | Always exist |
| Impredicativity | Yes | No (predicative) |

### 15.2 Let-Polymorphism

The key restriction in HM is **let-polymorphism**: polymorphic types (type schemes σ = ∀α₁...∀αₙ. τ) are only introduced by let-bindings, not by lambda abstractions.

```
let id = λx. x in (id 3, id true)    -- OK: id is polymorphic
(λid. (id 3, id true)) (λx. x)       -- Error: id is monomorphic
```

In the second example, the lambda-bound variable `id` must have a single monomorphic type, so it cannot be used at both `Int → Int` and `Bool → Bool`.

### 15.3 Algorithm W

Damas and Milner (1982) proved that HM has **principal types** (every typable term has a most general type) and gave **Algorithm W**, a complete type inference algorithm based on first-order unification. This algorithm runs in nearly-linear time in practice (though it is DEXPTIME-complete in the worst case due to the possibility of exponentially large principal types).

### 15.4 The Design Tradeoff

The HM restriction represents a carefully chosen point in the design space:
- It retains enough expressiveness for most practical functional programming.
- It provides complete type inference without annotations.
- It has principal types, enabling modular type checking.
- It sacrifices the ability to pass polymorphic functions as arguments (no rank-2+ polymorphism) and impredicative instantiation.
