# The Simply Typed Lambda Calculus

## Overview

The simply typed lambda calculus (STLC), also written as lambda-arrow or
lambda(->), is the most fundamental typed lambda calculus. It extends the
untyped lambda calculus by assigning simple types to every term, ensuring that
function application respects type discipline. The STLC occupies a central
position in the theory of programming languages: it is the kernel of every
typed functional language, provides the computational content of the
implicational fragment of intuitionistic propositional logic via the
Curry-Howard correspondence, and serves as the internal language of cartesian
closed categories via the Curry-Howard-Lambek correspondence.

The key metatheoretic properties of STLC are type safety (progress and
preservation), confluence, and strong normalization. The last of these means
that every well-typed term reduces to a (unique) normal form in finitely many
steps -- equivalently, every STLC program terminates. This makes STLC
Turing-incomplete, but also makes it a useful foundation for total functional
programming and proof assistants.

## 1. Historical Context

### Church (1940)

Alonzo Church introduced the simply typed lambda calculus in his 1940 paper
"A Formulation of the Simple Theory of Types" (Journal of Symbolic Logic 5(2),
pp. 56-68). Church had originally developed the untyped lambda calculus in
1932-1933 as a foundation for mathematics, but Kleene and Rosser (1935) showed
that the untyped system was logically inconsistent via a diagonal argument
(distinct from Russell's paradox). Church then developed the typed version
to restore consistency.

Church's type system assigns every expression a type drawn from a simple
hierarchy:
- iota (i): the type of individuals
- omicron (o): the type of truth values (propositions)
- (alpha beta): the type of functions from type beta to type alpha

Note that Church's arrow convention is reversed compared to modern usage.
In modern notation, Church's (alpha beta) would be written beta -> alpha.

Church adopted the method of currying (due to Schoenfinkel 1924 and Curry),
reducing n-ary functions to iterated unary functions.

### Curry (1958)

Haskell Curry, together with Robert Feys, observed in *Combinatory Logic,
Volume I* (1958) that the types assignable to combinators in typed combinatory
logic correspond exactly to tautologies of the implicational fragment of
intuitionistic propositional logic. This was the first glimpse of what would
become the Curry-Howard correspondence.

### Howard (1969/1980)

William Howard made the correspondence fully explicit in his 1969 manuscript
"The Formulae-as-Types Notion of Construction" (circulated informally,
published in 1980 in *To H.B. Curry: Essays on Combinatory Logic, Lambda
Calculus and Formalism*, Academic Press, pp. 479-490). Howard showed that:
- Types in STLC correspond to propositions in intuitionistic natural deduction
- Terms correspond to proofs
- Beta-reduction corresponds to proof normalization (cut elimination)

### de Bruijn (1970)

Nicolaas Govert de Bruijn independently discovered the propositions-as-types
correspondence in the context of his Automath proof checker (1968-1970). He
also introduced the nameless variable representation now called de Bruijn
indices (1972).

## 2. Syntax

### 2.1 Types

The types of STLC are generated by the following grammar:

```
T, S, U ::= B            -- base type (one or more atomic types)
           | T -> S       -- function (arrow) type
```

The arrow type constructor is right-associative:
  T1 -> T2 -> T3  means  T1 -> (T2 -> T3)

Common choices for the set of base types:
- A single base type B (minimal STLC)
- Bool with constants true, false and a conditional
- Nat with zero, successor, and a recursor (Goedel's System T)

### 2.2 Terms

```
e, f, g ::= x              -- variable
           | \x:T. e        -- lambda abstraction (with type annotation)
           | e1 e2          -- application
```

Application is left-associative:
  e1 e2 e3  means  (e1 e2) e3

In the Church-style (explicitly typed) presentation, lambda abstractions carry
type annotations on the bound variable. In the Curry-style (implicitly typed)
presentation, annotations are omitted: \x. e.

### 2.3 Typing Contexts

A typing context (or environment) Gamma is a finite ordered list of variable-type
bindings:

```
Gamma ::= .                 -- empty context
         | Gamma, x : T    -- extension with a fresh variable
```

We write x : T in Gamma to mean that the pair (x, T) appears in Gamma.
Contexts are subject to a freshness condition: each variable appears at most once.

### 2.4 Values and Neutral Terms

Values (fully reduced terms):
```
v ::= \x:T. e              -- lambda abstractions are values
```

(In extensions with base types, constants like true, false, 0 are also values.)

Neutral terms (terms that cannot reduce because they are "stuck" on a variable):
```
n ::= x | n e
```

### 2.5 Free and Bound Variables

The lambda abstraction \x:T. e binds x in e. The set of free variables:
```
FV(x)       = {x}
FV(\x:T. e) = FV(e) \ {x}
FV(e1 e2)   = FV(e1) union FV(e2)
```

A term is closed if FV(e) = {}.

## 3. Typing Rules

The typing judgment has the form:

  Gamma |- e : T

meaning "under context Gamma, term e has type T."

### Inference Rules

```
                x : T in Gamma
  (Var)     ----------------------
                Gamma |- x : T


              Gamma, x : T |- e : S
  (Abs)   ----------------------------
            Gamma |- (\x:T. e) : T -> S


            Gamma |- e1 : T -> S    Gamma |- e2 : T
  (App)   ----------------------------------------------
                      Gamma |- e1 e2 : S
```

These three rules completely determine the typing relation for the pure STLC.

### Properties of the Typing Relation

- **Uniqueness of types (Church-style):** If Gamma |- e : T and Gamma |- e : S,
  then T = S. Each term has at most one type in a given context.
- **Decidability:** Type checking is decidable (and efficient -- linear in the
  size of the term for Church-style).

## 4. Operational Semantics

### 4.1 Beta-Reduction

The fundamental computation rule:

  (\x:T. e) v  -->  [x := v] e     (beta-reduction, call-by-value)

where [x := v] e denotes the capture-avoiding substitution of v for x in e.

### 4.2 Capture-Avoiding Substitution

```
[x := s] x       = s
[x := s] y       = y                              (y /= x)
[x := s] (e1 e2) = ([x := s] e1) ([x := s] e2)
[x := s] (\y:T. e) = \y:T. [x := s] e            (y /= x, y not in FV(s))
```

The side condition on the abstraction case prevents variable capture. If y is
free in s, we alpha-convert the abstraction first: rename y to a fresh variable z.

### 4.3 Reduction Strategies

**Full beta-reduction:** Reduce any redex anywhere.

**Call-by-value (CBV):**
```
         e1 --> e1'
  -------------------------
     e1 e2 --> e1' e2

         e2 --> e2'
  -------------------------    (where e1 is a value)
     v e2 --> v e2'

  (\x:T. e) v --> [x := v] e
```

**Call-by-name (CBN):**
```
         e1 --> e1'
  -------------------------
     e1 e2 --> e1' e2

  (\x:T. e) e2 --> [x := e2] e
```

Since STLC is strongly normalizing (and confluent), all strategies reach the
same normal form.

## 5. Type Inference and Type Checking

### 5.1 Type Checking (Church-style)

For explicitly typed terms, type checking is straightforward:

```
check(Gamma, x, T)         = (lookup(Gamma, x) == T)
check(Gamma, \x:S. e, T)   = (T == S -> U) and check(Gamma,x:S, e, U)
check(Gamma, e1 e2, T)     = exists S. check(Gamma, e1, S -> T)
                                    and check(Gamma, e2, S)
```

This runs in time linear in the size of the term.

### 5.2 Type Inference (Curry-style)

For implicitly typed terms, type inference is more interesting. The algorithm
proceeds by:

1. Assign fresh type variables to unannotated positions.
2. Generate constraints by walking the term.
3. Solve constraints by unification.

This is essentially a restriction of the Hindley-Milner algorithm (Algorithm W)
to the simply typed case (no let-polymorphism).

**Algorithm (sketch):**
```
infer(Gamma, x) = lookup(Gamma, x)

infer(Gamma, e1 e2) =
  let T1 = infer(Gamma, e1)
  let T2 = infer(Gamma, e2)
  let alpha = fresh type variable
  unify(T1, T2 -> alpha)
  return alpha

infer(Gamma, \x. e) =
  let alpha = fresh type variable
  let T = infer(Gamma + {x : alpha}, e)
  return alpha -> T
```

**Key result (Hindley 1969):** The Curry-style STLC has the principal type
property -- if a term is typable, it has a most general type from which all
other valid types can be obtained by substitution of type variables. The
inference algorithm computes this principal type.

### 5.3 Bidirectional Type Checking

Bidirectional type checking splits the typing judgment into two modes:

- **Synthesis (inference):** Gamma |- e => T (the type is an output)
- **Checking:** Gamma |- e <= T (the type is an input)

Rules:
```
              x : T in Gamma
  (Var)     --------------------
             Gamma |- x => T

           Gamma |- e1 => T -> S    Gamma |- e2 <= T
  (App)   ------------------------------------------------
                    Gamma |- e1 e2 => S

              Gamma, x : T |- e <= S
  (Abs)   ------------------------------
           Gamma |- (\x. e) <= T -> S

             Gamma |- e => T
  (Sub)   ----------------------    (T = S)
             Gamma |- e <= S

              Gamma |- (e : T) => T    -- type annotation
  (Ann)   --------------------------------
```

Key properties:
- Lambda abstractions are checked (type flows in from the expected type)
- Variables and applications synthesize their types
- Annotations are needed only at beta-redexes
- This approach is simpler than full Hindley-Milner inference and scales better
  to richer type systems (dependent types, higher-rank polymorphism)

Reference: Dunfield, Jana and Krishnaswami, Neel. "Complete and Easy
Bidirectional Typechecking for Higher-Rank Polymorphism." ICFP, 2013.

## 6. Structural Properties

The STLC typing judgment satisfies several important structural properties,
which can be proved by straightforward inductions on typing derivations.

### 6.1 Weakening

**Theorem (Weakening):** If Gamma |- e : T and Gamma is a subset of Gamma'
(i.e., every binding in Gamma also appears in Gamma'), then Gamma' |- e : T.

*Proof:* By induction on the derivation of Gamma |- e : T. The variable case
follows from the subset inclusion. The abstraction case follows by extending
both contexts. The application case follows directly by IH.

Weakening says we can add unused hypotheses to the context without affecting
typability.

### 6.2 Exchange

**Theorem (Exchange):** If Gamma, x:S, y:T, Delta |- e : U, then
Gamma, y:T, x:S, Delta |- e : U.

*Proof:* By induction on the typing derivation. The variable case follows
because membership in the context is insensitive to ordering.

Exchange says the order of assumptions does not matter.

### 6.3 Contraction

**Theorem (Contraction):** If Gamma, x:T, y:T |- e : S, then
Gamma, z:T |- [x:=z, y:=z]e : S.

Contraction says we can merge two assumptions of the same type.

### 6.4 Substitution Lemma

**Theorem (Substitution):** If Gamma, x:U |- e : T and Gamma |- v : U,
then Gamma |- [x := v]e : T.

*Proof:* By induction on the structure of e.

This is the key lemma for proving preservation (see Section 7).

### 6.5 Substructural Variants

Substructural type systems arise by restricting one or more of the above
structural rules:

| System   | Weakening | Exchange | Contraction | Variable usage    |
|----------|-----------|----------|-------------|-------------------|
| STLC     | Yes       | Yes      | Yes         | Unrestricted      |
| Linear   | No        | Yes      | No          | Exactly once      |
| Affine   | Yes       | Yes      | No          | At most once      |
| Relevant | No        | Yes      | Yes         | At least once     |
| Ordered  | No        | No       | No          | Exactly once, in order |

- **Linear types** (Girard 1987, Wadler 1990) model resources that must be used
  exactly once. The linear lambda calculus is the term calculus for linear logic.
- **Affine types** allow discarding but not duplicating. Used in Rust's ownership
  system.
- **Ordered types** (Lambek calculus) are connected to non-commutative logic and
  have applications in linguistics (Lambek grammars).

## 7. Type Safety: Progress and Preservation

Type safety is the metatheoretic property that well-typed programs "don't go
wrong" -- they never reach a stuck state. The modern syntactic approach, due
to Wright and Felleisen (1994), establishes type safety via two lemmas.

### 7.1 Canonical Forms

**Lemma (Canonical Forms):**
- If |- v : T1 -> T2 and v is a value, then v = \x:T1. e for some x, e.
- If |- v : Bool and v is a value, then v = true or v = false.

*Proof:* By case analysis on the value predicate and inversion on the typing
derivation.

### 7.2 Progress

**Theorem (Progress):** If |- e : T (e is a closed, well-typed term), then
either:
  (a) e is a value, or
  (b) there exists e' such that e --> e'.

*Proof:* By induction on the typing derivation.
- Var case: impossible (the context is empty).
- Abs case: e = \x:T. body is already a value.
- App case: e = e1 e2. By IH, either e1 is a value or can step.
  If e1 can step, use the congruence rule.
  If e1 is a value, by IH either e2 is a value or can step.
  If e2 can step, use the congruence rule.
  If both are values, by canonical forms e1 = \x:T. body, and
  beta-reduction applies.

### 7.3 Preservation (Subject Reduction)

**Theorem (Preservation):** If |- e : T and e --> e', then |- e' : T.

*Proof:* By induction on the typing derivation.
- The critical case is beta-reduction: (\x:S. body) v --> [x := v] body.
  By inversion, we know |- \x:S. body : S -> T, which gives us
  x:S |- body : T. We also know |- v : S. By the Substitution Lemma,
  |- [x := v] body : T.

### 7.4 Type Soundness

**Corollary (Type Soundness):** If |- e : T and e -->* e' (multi-step
reduction to a normal form), then e' is a value of type T.

*Proof:* By induction on the number of steps, applying preservation at each
step to maintain well-typedness, then applying progress to the final term.
Since STLC is strongly normalizing, reduction always reaches a normal form.

## 8. Strong Normalization

### 8.1 Statement

**Theorem (Strong Normalization):** Every well-typed term in the simply typed
lambda calculus is strongly normalizing: every reduction sequence starting from
a well-typed term is finite.

Equivalently: there are no infinite reduction sequences from well-typed terms.
Since STLC is also confluent (Church-Rosser), every well-typed term has a
unique normal form, and all reduction strategies reach it.

### 8.2 Why Naive Induction Fails

A direct proof by induction on the typing derivation fails at the application
rule. If Gamma |- e1 e2 : T, and e1 = \x:S. body, then e1 e2 --> [x:=e2]body.
The substituted term [x:=e2]body may be larger than body (since e2 may be
substituted in multiple places), breaking the inductive measure.

### 8.3 Tait's Method: Reducibility / Logical Relations

The standard proof technique, introduced by Tait (1967), uses a logical
relation (also called "reducibility predicate" or "hereditary termination").

**Definition.** For each type T, define a set Red_T of terms (the "reducible"
or "strongly normalizable" terms) by induction on T:

- **Red_B(e)** iff |- e : B and e is strongly normalizing (SN).
- **Red_{S -> T}(e)** iff |- e : S -> T and e is SN, and for all e',
  if Red_S(e') then Red_T(e e').

**Key properties (Candidates of Reducibility conditions):**

- **CR1:** If Red_T(e), then e is SN.
  (Immediate from definition.)

- **CR2 (Backward closure):** If e --> e' and Red_T(e'), then Red_T(e).
  (By induction on T.)

- **CR3 (Neutral terms):** If e is neutral and all one-step reducts of e
  are in Red_T, then Red_T(e).
  (By induction on T.)

**Fundamental Lemma (Adequacy):** If Gamma |- e : T and sigma is a closing
substitution such that Red_{S_i}(sigma(x_i)) for each (x_i : S_i) in Gamma,
then Red_T(sigma(e)).

*Proof:* By induction on the typing derivation.
- Var: sigma(x) is reducible by assumption.
- Abs: \x:S. body. We must show for any Red_S(a), Red_T([x:=a](sigma(body))).
  Extend sigma with [x := a] and apply the IH.
- App: e1 e2. By IH, Red_{S->T}(sigma(e1)) and Red_S(sigma(e2)).
  By definition of Red_{S->T}, Red_T(sigma(e1) sigma(e2)).

**Corollary:** Every well-typed term is SN.
*Proof:* Apply the Fundamental Lemma with the identity substitution. Each
variable x_i is neutral, so by CR3 it is in Red_{S_i}. Hence e is in Red_T,
and by CR1, e is SN.

### 8.4 Girard's Reducibility Candidates

Girard (1972) generalized Tait's method to handle System F (polymorphic lambda
calculus). In Girard's approach, the logical relation is parameterized over
arbitrary sets of terms satisfying the CR conditions, called "reducibility
candidates" (candidats de reductibilite). For STLC, Tait's simpler approach
suffices.

### 8.5 Consequences of Strong Normalization

- STLC is not Turing-complete: it cannot express all computable functions.
  Specifically, pure STLC with Church numerals can only define the
  **extended polynomials** (a very restricted class of numeric functions).
- System T (STLC extended with natural numbers and primitive recursion;
  see Section 12.5) is significantly more expressive: its definable
  functions are exactly those provably total in Peano arithmetic.
  These are distinct systems and should not be conflated.
- Deciding beta-eta equality of STLC terms is decidable.

## 9. Curry-Howard Correspondence

The Curry-Howard correspondence (also called propositions-as-types,
proofs-as-programs, or the Brouwer-Heyting-Kolmogorov interpretation realized)
establishes a precise isomorphism between the STLC and the implicational
fragment of intuitionistic propositional logic.

### 9.1 The Correspondence Table

| Simply Typed Lambda Calculus  | Intuitionistic Natural Deduction           |
|-------------------------------|---------------------------------------------|
| Type T                        | Proposition T                               |
| Term e of type T              | Proof of proposition T                      |
| Arrow type A -> B             | Implication A => B                          |
| Lambda abstraction \x:A. e    | Implication introduction (assume A, prove B)|
| Application e1 e2             | Implication elimination (modus ponens)      |
| Variable x : A                | Use of hypothesis A                         |
| Typing context Gamma          | Set of assumptions / hypotheses             |
| Beta reduction                | Proof normalization / cut elimination       |
| Normal form                   | Normal proof (in the sense of Prawitz)      |
| Type inhabitation             | Provability                                 |
| Closed term of type T         | Proof of T from no assumptions (theorem)    |

### 9.2 Formal Statement

There is a bijection between:
- Typing derivations Gamma |- e : T in STLC
- Natural deduction proofs Gamma => T in the implicational fragment of
  intuitionistic propositional logic

Moreover, beta-reduction on terms corresponds exactly to Prawitz's proof
normalization (elimination of detours / cuts).

### 9.3 Examples

The type A -> A is inhabited by \x:A. x (the identity). This corresponds to
the trivial proof: assume A, conclude A.

The type A -> B -> A is inhabited by \x:A. \y:B. x (the K combinator). This
corresponds to the proof: assume A, assume B, conclude A (by the first
assumption).

The type (A -> B -> C) -> (A -> B) -> A -> C is inhabited by
\f. \g. \x. f x (g x) (the S combinator). This corresponds to a proof of
the corresponding tautology.

The type A -> B (with A, B distinct base types) is not inhabited. This
corresponds to the fact that A => B is not a tautology of propositional logic.

### 9.4 Consequences

- **Strong normalization of STLC** corresponds to the fact that every proof
  in the implicational fragment can be normalized (Prawitz 1965).
- **Type inhabitation for STLC** is equivalent to provability in the
  implicational fragment, which is PSPACE-complete (Statman 1979).
- **Decidability of type checking** corresponds to decidability of proof
  verification.

## 10. Categorical Semantics: Cartesian Closed Categories

### 10.1 Definition

A **cartesian closed category** (CCC) is a category C equipped with:

1. A terminal object 1 (corresponding to the unit type).
2. Binary products: for every pair of objects A, B, a product object A x B
   with projections pi1 : A x B -> A and pi2 : A x B -> B, and a pairing
   operation <f, g> : C -> A x B for any f : C -> A and g : C -> B.
3. Exponential objects: for every pair of objects A, B, an object B^A
   (also written A => B or [A, B]) with an evaluation morphism
   eval : B^A x A -> B, such that for every f : C x A -> B there is a
   unique curry(f) : C -> B^A with eval o (curry(f) x id_A) = f.

### 10.2 The Correspondence (Lambek 1980)

Lambek observed that the STLC (with products) and cartesian closed categories
are essentially the same thing, viewed from different angles:

**From STLC to CCC:**
Given a simply typed lambda calculus, construct a category C where:
- Objects = types
- Morphisms from A to B = equivalence classes of terms x:A |- e : B
  (up to beta-eta equality)
- Composition = substitution
- Identity = the variable term x:A |- x : A
- Terminal object = unit type
- Products = product types
- Exponentials = function types (currying = lambda abstraction)

This category is cartesian closed.

**From CCC to STLC:**
Given a CCC, construct a simply typed lambda calculus where:
- Base types = objects of the category
- Arrow types = exponential objects
- Terms in context Gamma |- e : T = morphisms from Gamma to T
- Lambda abstraction = currying
- Application = evaluation morphism

### 10.3 The Curry-Howard-Lambek Correspondence

The three-way correspondence:

| Lambda Calculus         | Logic                 | Category Theory          |
|-------------------------|-----------------------|--------------------------|
| Types                   | Propositions          | Objects                  |
| Terms Gamma |- e : T   | Proofs Gamma => T     | Morphisms Gamma -> T     |
| Function type A -> B    | Implication A => B    | Exponential B^A          |
| Product type A x B      | Conjunction A /\ B    | Product A x B            |
| Sum type A + B          | Disjunction A \/ B    | Coproduct A + B          |
| Unit type 1             | Truth (top)           | Terminal object 1        |
| Empty type 0            | Falsity (bottom)      | Initial object 0         |
| Beta-eta equality       | Proof equivalence     | Equality of morphisms    |
| Substitution            | Cut rule              | Composition              |
| Identity \x.x           | Axiom / hypothesis    | Identity morphism        |

### 10.4 Significance

- Every model of STLC in a CCC is automatically sound.
- The free CCC generated by a set of base types is the "syntactic category"
  of the corresponding STLC, and it is the initial model.
- This gives a canonical denotational semantics: the interpretation of a term
  is its equivalence class in the syntactic category.
- Practical applications: Conal Elliott's "Compiling to Categories" (ICFP 2017)
  uses this correspondence to compile Haskell programs to various CCCs
  (circuits, automatic differentiation, etc.).

Key references:
- Lambek, J. "From Lambda Calculus to Cartesian Closed Categories."
  In *To H.B. Curry*, Academic Press, 1980.
- Lambek, J. and Scott, P.J. *Introduction to Higher Order Categorical Logic*.
  Cambridge University Press, 1986.

## 11. Denotational Semantics

### 11.1 Set-Theoretic Semantics

The simplest denotational semantics for STLC interprets types as sets and
terms as functions between sets:

- [[B]] = some chosen set D_B (the domain of base type B)
- [[S -> T]] = [[T]]^[[S]] = the set of all functions from [[S]] to [[T]]

An environment rho maps variables to elements of their types' denotations.

- [[x]]_rho = rho(x)
- [[\x:S. e]]_rho = the function f where f(d) = [[e]]_{rho[x := d]}
- [[e1 e2]]_rho = [[e1]]_rho ([[e2]]_rho)

This interpretation is well-defined because STLC is strongly normalizing --
there are no divergent computations, so we never need to model non-termination.

**Soundness:** If Gamma |- e : T and rho models Gamma (i.e., rho(x) in [[S]]
for each x:S in Gamma), then [[e]]_rho is in [[T]].

**Completeness:** For the full set-theoretic model, there exist functions in
[[S -> T]] that are not definable by any STLC term. The model is not fully
abstract.

### 11.2 Domain-Theoretic Semantics

For extensions of STLC with recursion (e.g., PCF), a set-theoretic semantics
is inadequate because we must model non-termination. Dana Scott (1969-1970)
developed domain theory to address this:

- Types are interpreted as domains (directed-complete partial orders, dcpos,
  or more specifically, Scott domains or continuous lattices).
- Each domain has a bottom element (representing non-termination).
- Functions must be continuous (preserve directed suprema).
- [[S -> T]] = the set of continuous functions from [[S]] to [[T]].
- Recursive definitions are interpreted as least fixed points (by the
  Kleene fixed-point theorem / Scott's fixed-point theorem).

For the pure STLC without recursion, domain-theoretic and set-theoretic
semantics coincide (since there is no non-termination to model).

## 12. Extensions

### 12.1 Product Types (Pairs)

```
Types:     T ::= ... | T1 x T2 | 1
Terms:     e ::= ... | (e1, e2) | fst e | snd e | ()
```

Typing rules:
```
  Gamma |- e1 : T1    Gamma |- e2 : T2
  ----------------------------------------
        Gamma |- (e1, e2) : T1 x T2

     Gamma |- e : T1 x T2            Gamma |- e : T1 x T2
  -------------------------       -------------------------
   Gamma |- fst e : T1              Gamma |- snd e : T2

  -----------------------
    Gamma |- () : 1
```

Reduction rules:
```
  fst (v1, v2) --> v1        snd (v1, v2) --> v2
```

Eta law: (fst e, snd e) = e  (for e : T1 x T2)

The unit type 1 has a single value () and serves as the nullary product
(terminal object in the categorical semantics).

### 12.2 Sum Types (Coproducts)

```
Types:     T ::= ... | T1 + T2 | 0
Terms:     e ::= ... | inl e | inr e | case e of inl x => e1 | inr y => e2
                     | absurd e
```

Typing rules:
```
       Gamma |- e : T1                    Gamma |- e : T2
  -------------------------          -------------------------
  Gamma |- inl e : T1 + T2          Gamma |- inr e : T1 + T2

  Gamma |- e : T1 + T2    Gamma,x:T1 |- e1 : S    Gamma,y:T2 |- e2 : S
  -------------------------------------------------------------------------
       Gamma |- case e of inl x => e1 | inr y => e2 : S

      Gamma |- e : 0
  -----------------------
  Gamma |- absurd e : T       (ex falso quodlibet / principle of explosion)
```

The void type 0 has no values and corresponds to falsity in the Curry-Howard
correspondence. The elimination form absurd e corresponds to the logical
principle "from falsehood, anything follows."

### 12.3 Let-Bindings

```
Terms:     e ::= ... | let x = e1 in e2
```

Typing rule:
```
  Gamma |- e1 : S    Gamma, x:S |- e2 : T
  -------------------------------------------
       Gamma |- let x = e1 in e2 : T
```

Reduction: let x = v in e2 --> [x := v] e2

In the simply typed setting, let x = e1 in e2 is equivalent to (\x:S. e2) e1.
The distinction becomes important in Hindley-Milner, where let introduces
polymorphism.

### 12.4 Recursion (General Recursion via Fix)

```
Terms:     e ::= ... | fix e
```

Typing rule:
```
     Gamma |- e : T -> T
  --------------------------
    Gamma |- fix e : T
```

Reduction: fix (\x:T. body) --> [x := fix (\x:T. body)] body

**Warning:** Adding fix destroys strong normalization. The term
fix (\x:T. x) reduces to itself indefinitely. With fix, the calculus becomes
Turing-complete (this gives PCF when combined with natural numbers and
booleans). Type safety (progress + preservation) is preserved, but termination
is lost.

### 12.5 Natural Numbers (System T / Goedel)

```
Types:     T ::= ... | Nat
Terms:     e ::= ... | 0 | succ e | natrec e e_z (\x. \r. e_s)
```

The recursor natrec provides primitive recursion. System T (Goedel 1958) is
STLC extended with natural numbers and primitive recursion. It is still
strongly normalizing but can express all primitive recursive functions and
some that go beyond (e.g., the Ackermann function).

## 13. Relationship to Typed Combinatory Logic

### 13.1 Typed Combinators

The simply typed lambda calculus has a well-known correspondence with typed
combinatory logic. The basic typed combinators are:

```
I : A -> A                          I = \x:A. x
K : A -> B -> A                     K = \x:A. \y:B. x
S : (A -> B -> C) -> (A -> B) -> A -> C
                                    S = \f. \g. \x. f x (g x)
```

These combinators are typed instances of the SKI combinator basis.

### 13.2 Abstraction Elimination (Bracket Abstraction)

Any STLC term can be translated to an equivalent typed combinator expression
by the bracket abstraction algorithm:

```
[x] x       = I
[x] e       = K e           (x not free in e)
[x] (e1 e2) = S ([x] e1) ([x] e2)
```

where [x] e denotes the abstraction of x from e.

### 13.3 Properties

- The translation preserves types and semantics.
- The translation is not size-preserving: it can cause an exponential blowup.
  More efficient translations exist (using B, C, S', and other combinators).
- Under the Curry-Howard correspondence, the S, K, I combinators correspond to
  specific axiom schemes of the Hilbert-style presentation of intuitionistic
  implicational logic:
  - K corresponds to the axiom A => B => A
  - S corresponds to the axiom (A => B => C) => (A => B) => (A => C)
  - I corresponds to A => A (derivable from S and K)

## 14. De Bruijn Representation

### 14.1 Motivation

Named variables require alpha-conversion to identify terms that differ only
in the choice of bound variable names. De Bruijn indices provide a canonical
representation where alpha-equivalent terms are syntactically identical.

### 14.2 Syntax

```
Types:     T ::= B | T1 -> T2
Contexts:  Gamma ::= . | Gamma, T       (a list of types, no variable names)
Terms:     e ::= n | \. e | e1 e2       (n is a natural number)
```

A de Bruijn index n refers to the variable bound by the (n+1)-th enclosing
lambda, counting from 0.

### 14.3 Examples

| Named term              | De Bruijn term     |
|--------------------------|--------------------|
| \x. x                   | \. 0               |
| \x. \y. x               | \. \. 1            |
| \x. \y. y               | \. \. 0            |
| \f. \x. f x             | \. \. 1 0          |
| \f. \g. \x. f x (g x)   | \. \. \. 2 0 (1 0)|

### 14.4 Typing Rules with De Bruijn Indices

Variables are typed by position in the context:
```
        Gamma = Gamma', T          (T is at position 0)
  ----------------------------------------
          Gamma |- 0 : T

        Gamma = Gamma', S          Gamma' |- n : T
  -------------------------------------------------------
          Gamma |- (n+1) : T

             Gamma, T |- e : S
  ----------------------------------------
          Gamma |- (\. e) : T -> S

       Gamma |- e1 : T -> S    Gamma |- e2 : T
  --------------------------------------------------
              Gamma |- e1 e2 : S
```

### 14.5 Substitution with Shifting

Substitution in de Bruijn notation requires a shifting operation to adjust
free variable indices when passing under binders:

```
shift(d, c, n) = n           if n < c    (bound variable, do not shift)
shift(d, c, n) = n + d       if n >= c   (free variable, shift by d)
shift(d, c, \. e) = \. shift(d, c+1, e)
shift(d, c, e1 e2) = shift(d, c, e1)  shift(d, c, e2)
```

Single substitution:
```
[k := s] k       = s
[k := s] n       = n                        (n /= k)
[k := s] (\. e)  = \. [k+1 := shift(1,0,s)] e
[k := s] (e1 e2) = ([k := s] e1) ([k := s] e2)
```

### 14.6 Intrinsically Typed De Bruijn Terms

In proof assistants (Agda, Lean, Coq), it is common to use an intrinsically
typed representation where the term datatype is indexed by both context and
type. This makes it impossible to even represent ill-typed terms, and
substitution is type-preserving by construction.

In Agda (from PLFA by Wadler and Kokke):
```agda
data _⊢_ : Context → Type → Set where
  `_   : Γ ∋ A → Γ ⊢ A                       -- variable (de Bruijn index)
  ƛ_   : Γ , A ⊢ B → Γ ⊢ A ⇒ B              -- abstraction
  _·_  : Γ ⊢ A ⇒ B → Γ ⊢ A → Γ ⊢ B          -- application
```

### 14.7 Advantages of De Bruijn Representation

1. Alpha-equivalence becomes syntactic equality.
2. No need for freshness conditions or alpha-conversion during substitution.
3. Contexts become simple lists of types (no variable names).
4. Natural fit for mechanized metatheory: many proofs become simpler.
5. Used extensively in implementations of proof assistants and type checkers.

## Cross-References

- **Doc 01 (Untyped Lambda Calculus):** The untyped lambda calculus is the untyped
  foundation from which STLC was derived. Church introduced STLC after the
  Kleene-Rosser paradox showed that the untyped system with logical axioms was
  inconsistent.

- **Doc 03 (System F):** System F extends STLC with parametric polymorphism
  (universal quantification over types), dramatically increasing expressive power.
  While pure STLC can only define extended polynomials, System F can express all
  functions provably total in second-order Peano arithmetic.

- **Doc 09 (Implementation / Abstract Machines):** Abstract machines provide
  concrete evaluation strategies for STLC and its extensions. The CEK machine,
  Krivine machine, and SECD machine formalize environment-based evaluation of
  the operational semantics described in Section 4.
